# Диагностика: на сервере применяется старая логика type_rules.yaml

## 1. Откуда загружается type_rules.yaml

- **Файл:** `app/infrastructure/config/loader.py`
- **Функция:** `load_type_rules(configs_dir=None)` → вызывает `load_yaml("type_rules", configs_dir)` → читает `{configs_dir}/type_rules.yaml`.
- **Путь при `configs_dir=None`:** используется глобальная переменная `_CONFIGS_DIR`, которая задаётся **один раз при первом импорте** модуля `loader`:
  ```python
  _CONFIGS_DIR = _detect_configs_dir()  # выполняется при import loader
  ```
- **Как вычисляется `_CONFIGS_DIR`** (`_detect_configs_dir()`):
  - `here = Path(__file__).resolve()` — путь к файлу `loader.py`.
  - Первый кандидат: `here.parent.parent.parent / "configs"`  
    (от `.../app/infrastructure/config/loader.py` → `.../app/configs`).
  - Далее для PyInstaller: `sys._MEIPASS/app/configs` и т.д.
  - Возвращается **первый существующий** каталог из списка.

Итог: фактический путь к конфигам = каталог `app/configs` **относительно того места, откуда загружен модуль `loader.py`** (т.е. от `__file__` пакета `app`).

---

## 2. Захардкожен ли путь к другому конфигу

- В коде **нет** захардкоженного альтернативного пути к другому `type_rules.yaml`.
- Единственный явный путь к конфигам в скриптах: в `scripts/diagnose_redirect.py` задаётся `configs_dir = PROJECT_ROOT / "app" / "configs"` — это явное указание на репозиторий, не на тип-правила.

---

## 3. Кеширование

- В коде приложения **нет** `st.cache`, `st.cache_data` или другого кеша для конфигов или правил типов.
- `load_type_rules()` при каждом вызове заново читает YAML с диска (через `load_yaml`), но **каталог** берётся из `_CONFIGS_DIR`, который задаётся один раз при импорте.

---

## 4. Загрузка при старте и обновление

- Конфиг **не** загружается один раз при старте и не хранится в глобальной переменной; при каждом формировании отчёта вызывается `load_type_rules(configs_dir)` (с `configs_dir=None` в Streamlit).
- **Жёстко зафиксирован только путь к каталогу** — `_CONFIGS_DIR` вычисляется при первом импорте `loader.py`. Если этот путь указывает на **установленный пакет** (например, в `site-packages`), то после `git pull` обновляются файлы в репозитории, но не в установленной копии пакета — и приложение продолжает читать старый `type_rules.yaml` из установленного пакета.

---

## 5. Несколько файлов type_rules.yaml

- В проекте найден **один** файл: `app/configs/type_rules.yaml`. Дубликатов нет.

---

## 6. Использование dist/ или другой сборки вместо app/

- В `loader.py` есть поддержка PyInstaller (`sys.frozen`, `sys._MEIPASS`) — при запуске из билда конфиги ищутся в распакованном каталоге билда.
- Если на сервере **не** используется билд, но приложение запускается так, что пакет `app` подхватывается из **установки** (например, `pip install .` в venv или системный site-packages), то `__file__` у `loader.py` будет внутри этой установки (например, `/root/avito-report-builder-app/venv/lib/python3.x/site-packages/app/infrastructure/config/loader.py` или аналогичный путь). Тогда `_CONFIGS_DIR` = `.../site-packages/app/configs` — это **снимок конфигов на момент установки**. После `git pull` и `systemctl restart` обновляется только репозиторий в `WorkingDirectory`, а не файлы внутри установленного пакета, поэтому на сервере продолжает применяться старая логика.

---

## Вывод: возможная причина расхождения

- **Локально:** обычно запуск из корня репозитория (`streamlit run app.py`), в `sys.path` первым идёт проект, пакет `app` грузится из `./app`, `__file__` в loader — из репозитория, значит `_CONFIGS_DIR` = `./app/configs` → читается актуальный `type_rules.yaml`.
- **На сервере:** при запуске через systemd (например, из того же каталога) порядок путей в `sys.path` или способ запуска (например, через `python -m streamlit run app`) может привести к тому, что подхватывается **установленный** пакет `app` (из venv или site-packages). Тогда `_CONFIGS_DIR` указывает на конфиги внутри установки, которые не обновляются при `git pull`.

---

## Рекомендуемое исправление (минимальное)

Не менять архитектуру загрузки конфигов, а гарантировать, что **Streamlit-приложение всегда использует конфиги из каталога репозитория**, где лежит `app.py`:

- В **точке входа** (например, `app.py`) вычислить каталог конфигов относительно корня проекта:  
  `configs_dir = Path(__file__).resolve().parent / "app" / "configs"`.
- Передавать этот `configs_dir` во все вызовы `generate_report(..., configs_dir=configs_dir)`.

Тогда при любом способе установки и запуска (репозиторий, venv, systemd) конфиги будут читаться из одного и того же места — рядом с запускаемым `app.py` (репозиторий в `WorkingDirectory`), и после `git pull` и перезапуска сервиса будет применяться обновлённая логика `type_rules.yaml`.
