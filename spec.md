Техническое задание: Avito Report Builder
1. Product Overview
Avito Report Builder — внутренний web-инструмент агентства для автоматической обработки Excel-выгрузок из Авито и формирования стандартизированного Excel-отчёта.
Для кого
Сотрудники агентства, которые регулярно готовят отчёты по рекламным размещениям на Авито.
Какую задачу решает
Система заменяет ручную обработку Excel-выгрузок:
* удаление лишних столбцов;
* переименование полей;
* очистку данных;
* расчёт производных показателей;
* сборку итоговой Excel-книги;
* оформление итогового отчёта.
Главная ценность
* сокращение времени подготовки отчётов;
* снижение количества ручных ошибок;
* единый стандарт итогового отчёта;
* воспроизводимый процесс для нескольких сотрудников.
Формат
Внутреннее web-приложение.

2. Scope and Boundaries
Что система делает
* принимает один .xlsx-файл;
* валидирует его структуру;
* читает исходный лист;
* нормализует и очищает данные;
* приводит входные колонки к внутренней канонической схеме;
* вычисляет производные поля;
* строит листы итогового отчёта;
* оформляет итоговую Excel-книгу;
* отдает пользователю итоговый .xlsx на скачивание.
Что система не делает
* не хранит файлы и историю обработки;
* не имеет БД;
* не поддерживает аутентификацию и роли;
* не позволяет редактировать правила через UI;
* не загружает данные из Авито напрямую;
* не сравнивает периоды;
* не обрабатывает несколько файлов одновременно;
* не строит аналитику по историческим данным.
Что входит в MVP
* Streamlit UI для загрузки файла и скачивания результата;
* валидация входного Excel;
* pipeline чтения, нормализации, enrichment;
* классификация Тип;
* сборка листов Админ панель, Диаграмма, Недельный, Сводная;
* расчёт итоговых метрик;
* экспорт Excel с базовым форматированием;
* понятные ошибки пользователю.
Что вне текущей версии
* API;
* редактирование правил через интерфейс;
* хранение истории генераций;
* несколько шаблонов отчёта;
* многопользовательские настройки;
* справочники и master-data;
* внешние интеграции;
* деплой в корпоративную инфраструктуру как отдельный контур;
* мультифайловая обработка.

3. User Roles
Role: User
Кто это: сотрудник агентства, формирующий отчёт.
Что делает:
* открывает приложение;
* загружает Excel-файл;
* запускает обработку;
* скачивает итоговый файл;
* читает сообщения об ошибках.
Ключевые действия:
* upload source report;
* start report generation;
* download result;
* при необходимости ввести Кабинет Авито вручную.
Уровень доступа:
* полный доступ ко всем функциям MVP;
* разграничение прав отсутствует.
Role: System
Кто это: приложение как исполняющий механизм pipeline.
Что делает:
* валидирует;
* трансформирует;
* вычисляет;
* строит листы;
* экспортирует workbook;
* сообщает ошибки.
Уровень доступа:
* внутренний системный исполнитель, не пользовательская роль.

4. Core User Flows
Flow 1. Генерация отчёта
Кто выполняет: User
Шаги:
1. Пользователь открывает web-приложение.
2. Загружает .xlsx файл.
3. При необходимости указывает Кабинет Авито.
4. Нажимает кнопку обработки.
5. Система валидирует файл.
6. Система читает и нормализует данные.
7. Система формирует итоговые листы.
8. Система собирает и форматирует Excel-книгу.
9. Пользователь скачивает итоговый файл.
Результат:Пользователь получает готовый .xlsx в фиксированном шаблоне.
Ограничения и проверки:
* разрешён только один файл;
* формат только .xlsx;
* входной шаблон должен соответствовать ожидаемой структуре;
* при критической ошибке обработка прерывается.
Flow 2. Ошибка валидации входного файла
Кто выполняет: User + System
Шаги:
1. Пользователь загружает файл.
2. Система определяет структурную или файловую ошибку.
3. Обработка не запускается или прерывается.
4. Пользователю показывается понятное сообщение.
Результат:Отчёт не генерируется, пользователь понимает причину.
Ограничения и проверки:
* ошибки должны быть человекочитаемыми;
* для критических ошибок итоговый файл не формируется.
Flow 3. Обработка файла с частично плохими данными
Кто выполняет: System
Шаги:
1. Система читает файл.
2. Находит частично пустые или неидеальные значения.
3. Для некритичных кейсов применяет безопасные преобразования.
4. Продолжает обработку.
5. Формирует итоговый отчёт.
Результат:Отчёт формируется, если проблемы не ломают структуру.
Важные ограничения и проверки:
* безопасные значения и поведение должны быть зафиксированы;
* требуется единая политика для пустых чисел, деления на ноль и невалидных значений.

5. Core Entities
SourceReport
Исходный Excel-файл выгрузки Авито.
SourceSheet
Лист исходной выгрузки внутри входного workbook.Предположение: входной файл содержит один целевой лист данных и, возможно, служебные листы.
NormalizedRow
Одна строка после очистки, приведения типов и маппинга к внутренним именам полей.
ReportDataset
Единый нормализованный набор данных, на котором строятся все итоговые листы.
WeeklySheetData
Подготовленные данные для листа Недельный.
ChartSheetData
Подготовленные данные для листа Диаграмма.
AdminSheetData
Подготовленные данные для листа Админ панель.
SummarySheetData
Агрегированные показатели для листа Сводная.
ReportWorkbook
Итоговая Excel-книга на экспорт.
TypeRule
Правило классификации заголовка объявления в значение Тип.
ColumnMapping
Правило сопоставления входной колонки с внутренним каноническим полем.
WorkbookLayout
Конфигурация структуры итоговой книги: порядок листов, колонки, форматирование.

8. Statuses and Lifecycle
В системе нет пользовательских процессных сущностей с долгим жизненным циклом, но есть технический lifecycle генерации отчёта.
Entity: Report Generation Job
Предположение: отдельная сущность в БД не создаётся, но жизненный цикл важен для orchestration.
Статусы
* uploaded
* validating
* validation_failed
* reading
* normalizing
* enriching
* building_sheets
* writing_workbook
* formatting_workbook
* completed
* failed
Допустимые переходы
* uploaded → validating
* validating → validation_failed
* validating → reading
* reading → normalizing
* normalizing → enriching
* enriching → building_sheets
* building_sheets → writing_workbook
* writing_workbook → formatting_workbook
* formatting_workbook → completed
* любой активный этап → failed
Кто меняет статус
* только система
Что происходит при смене статуса
* логируется текущий этап;
* при validation_failed пользователю выводится ошибка структуры;
* при failed пользователю выводится общая ошибка обработки;
* при completed появляется ссылка/кнопка скачивания файла.

9. Business Rules
9.1 Общие правила
* Система принимает только один .xlsx файл за один запуск.
* Входной формат один и фиксированный.
* Выходной формат один и фиксированный.
* Приложение является шаблонным генератором отчёта, а не универсальным Excel-конструктором.
* Все расчёты Сводная выполняются только по текущему входному файлу.
* История не хранится.
9.2 Валидация файла
* Файл обязан быть Excel-книгой .xlsx.
* Файл не должен быть пустым.
* Должен существовать ожидаемый лист с данными.
* Должны присутствовать обязательные колонки.
* При критическом несоответствии обработка останавливается.
9.3 Нормализация данных
* Пустые полностью строки удаляются.
* Заголовки колонок нормализуются и сопоставляются внутренним именам.
* Текстовые поля очищаются от лишних пробелов.
* Числовые поля приводятся к числовому типу.
* Даты приводятся к согласованному формату даты.
* Неожиданные дополнительные колонки игнорируются, если не ломают обработку.Предположение
9.4 Правила поля Тип
* Значение вычисляется по title.
* Правила должны храниться отдельно от основного кода.
* Допустимый источник правил: YAML-конфиг.
* При конфликте нескольких правил применяется правило с наивысшим приоритетом.Предположение
* Если совпадение не найдено, присваивается безопасное значение.Варианты:
    1. Не определено
    2. пустое значениеПредпочтительно для MVP: Не определено
9.5 Правила поля Кабинет Авито
Точное правило не зафиксировано.
Варианты реализации для MVP:
1. поле ввода в UI перед генерацией;
2. константа в конфиге.
Предпочтительно: поле ввода в UI, с возможностью дефолта из конфига.
9.6 Правила поля Номер на Авито
* Заполняется из исходного поля номера объявления.
* Должно сохранять значение, пригодное для вывода в итоговом листе.
* Форматирование значения должно быть устойчиво к потере ведущих символов, если они значимы.Предположение: хранить как string.
9.7 Правила поля Куда переадресация
* Заполняется через явное сопоставление по полю employee или другому полю источника.
* Правило не должно быть зашито хаотично в sheet builder.
* Источник сопоставления должен быть конфигурируемым.
Варианты реализации:
1. YAML-словарь employee -> redirect_target;
2. простая функция-mapper в коде.Предпочтительно: YAML-словарь.
9.8 Правила листа Сводная
* Все метрики считаются только по текущему файлу.
* Суммы считаются по соответствующим колонкам.
* конверсия = contacts * 100 / views
* средняя стоимость заявки = ad_spend / contactsТребует уточнения: использовать только ad_spend или общий расход по всем spend-полям.
* При делении на ноль используется безопасное значение.Предпочтительно: 0.
* Поле Комментарий должно быть пустым в сгенерированном отчёте.
9.9 Обработка невалидных данных
* Критичные структурные ошибки → остановка.
* Некритичные ошибки значений → обработка продолжается.
* Для пустых числовых значений используется единая политика дефолта.Предпочтительно: 0 для агрегируемых метрик.
* Частично неполные строки либо:
    1. остаются в датасете с дефолтами;
    2. отбрасываются, если нарушают обязательный минимум.Предпочтительно: оставлять, если есть ad_id или title, иначе удалять.Предположение
9.10 Удаление / архивирование
* Удаление сущностей и архивирование в MVP отсутствуют.
* Файлы не сохраняются на сервере после ответа пользователю.Предположение: обработка в памяти или во временном файле.

12. API Draft
Для MVP внешний HTTP API не обязателен, но ниже — draft внутреннего сервиса и возможного будущего API.
12.1 Внутренний application API
generate_report
* method: function call
* path: n/a
* purpose: полный orchestration генерации отчёта
* input: input_bytes: bytes, cabinet_name: str | None = None
* output: bytes итогового .xlsx
* access role: User via UI
validate_workbook
* method: function call
* purpose: проверить структуру файла до обработки
* input: input_bytes: bytes
* output: validation result / exception
* access role: System
12.2 Возможный future HTTP API
POST /api/v1/report/generate
* purpose: сгенерировать отчёт из входного файла
* input: multipart file + optional cabinet_name
* output: binary .xlsx
* access role: internal user / service
POST /api/v1/report/validate
* purpose: проверить файл без генерации
* input: multipart file
* output: JSON c результатом валидации и списком ошибок
* access role: internal user / service

14. Error Cases and Edge Cases
Файловые ошибки
* загружен не .xlsx;
* файл пустой;
* файл повреждён;
* книга не читается библиотекой.
Ошибки структуры
* отсутствует нужный лист;
* обязательные колонки отсутствуют;
* названия колонок отличаются сильнее, чем допускает mapping;
* шапка находится не в ожидаемой строке.Требует уточнения
Ошибки данных
* в числовых полях текст;
* пустые значения в обязательных колонках;
* даты в неожиданном формате;
* дубли строк / дубли объявлений;
* отрицательные расходы или метрики.Требует уточнения, допустимо ли это
* деление на ноль в расчётах;
* нераспознанный title для классификации Тип.
Конфликтные кейсы
* один заголовок подходит под несколько правил Тип;
* значение employee отсутствует, а redirect_target обязателен;
* cabinet_name не передан, а в шаблоне отчёта поле должно быть заполнено.
Рекомендуемое поведение
* критические ошибки структуры — stop processing;
* некритичные ошибки значений — warning + safe defaults;
* нераспознанный Тип — безопасное значение;
* деление на ноль — 0;
* неизвестный сотрудник для redirect_target:
    * вариант 1: пустое значение;
    * вариант 2: Не определеноПредпочтительно: пустое значение + warning.

15. Non-Functional Requirements
Производительность
* обработка одного стандартного Excel-файла должна занимать секунды;
* целевое время для типового файла: до 5 секунд.Предположение, требует проверки на реальных объёмах.
Масштабируемость
* горизонтальная масштабируемость не является требованием MVP;
* архитектура должна позволять вынести логику в API/сервис позже.
Безопасность
* только внутреннее использование;
* загрузка только .xlsx;
* не выполнять макросы и внешние формулы;
* не хранить файлы дольше времени сессии.Предположение
* требуется базовая защита от аварий на невалидных файлах.
Логирование
* логировать этапы pipeline;
* логировать причины ошибок в технический лог;
* в UI показывать упрощённые сообщения без stack trace.
Мобильная адаптация
* не обязательна;
* desktop-first интерфейс.
Отказоустойчивость
* при ошибке одного файла приложение должно продолжать работать для следующей загрузки;
* ошибка обработки не должна приводить к падению сервиса.
Хранение файлов
* постоянное хранение не требуется;
* допустим временный storage в памяти или temp directory.
История изменений
* история генераций в MVP отсутствует;
* versioning конфигов рекомендуется вести через git.
Тестируемость
* модульные тесты для validators, classifiers, transformers, sheet_builders, summary_calculator;
* fixture-based tests на эталонных Excel.
Поддерживаемость
* бизнес-правила должны быть отделены от Excel formatting;
* конфиги — отдельно от кода;
* orchestration — отдельно от domain logic.

16. MVP Scope
MVP
* один входной .xlsx;
* один фиксированный шаблон входа;
* один фиксированный выходной шаблон;
* Streamlit UI;

* классификация Тип;

    * Недельный
    * Сводная
* расчёт сводных метрик;
* форматирование workbook;
* скачивание результата;
* понятные ошибки.
Later
* несколько шаблонов входных файлов;
* несколько видов отчётов;
* редактирование правил через UI;
* отдельный API;
* хранение истории генераций;

Чтобы не перегрузить первую версию, имеет смысл жёстко зафиксировать:
* один входной лист;
* минимум ручных полей;

* только обязательное форматирование без сложных диаграмм Excel, если они не критичны.Риск перегруза MVP: лист Диаграмма может потребовать отдельного уточнения, что именно на нём должно быть.

18. Risks / Assumptions / Open Questions
Risks
1. Неявные ручные правила обработки ещё не полностью выявлены.
2. Классификация Тип по заголовку может оказаться нестабильной без набора синонимов и исключений.
3. Изменение шаблона выгрузки Авито сломает validation/mapping.
4. Требования к итоговому Excel-оформлению могут оказаться сложнее, чем кажется из описания.
5. Лист Диаграмма не описан содержательно и может потребовать отдельной логики.
6. Неопределённость по расходам в Сводная может привести к неверным бизнес-метрикам.
Assumptions
1. Входной файл — всегда .xlsx, без необходимости поддерживать .xls и .csv.
2. Во входном файле есть один основной лист данных.
3. Заголовки находятся в фиксированной строке.
4. ad_id и title достаточно для минимальной идентификации строки.
5. cabinet_name можно передавать из UI.
6. redirect_target можно заполнять через конфиг-маппинг.
7. Основные расчёты делаются в Python, а не формулами Excel.
8. Сервис может работать без постоянного хранения файлов и данных.

6. Какое точное правило расчёта общих расходов:
