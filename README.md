# Avito Report Builder

Внутренний инструмент для обработки Excel-выгрузок Авито и формирования стандартизированного отчёта в формате, максимально близком к эталонному `expected_report.xlsx`.

## Что умеет MVP

- Читает один входной файл `input_example.xlsx` (выгрузка Авито, лист `Sheet1`, 30 колонок).
- Валидирует структуру: формат `.xlsx`, наличие листа `Sheet1`, наличие обязательных колонок (см. `data/input_structure.md` и `app/configs/input_schema.yaml`).
- Нормализует данные:
  - приводит заголовки к каноническим именам (по `column_mapping.yaml`);
  - извлекает текст и ID из формул `HYPERLINK(...)` для номера и названия объявления;
  - чистит строки (strip, безопасные numeric/date-преобразования);
  - удаляет полностью пустые строки.
- Обогащает датасет:
  - вычисляет `type` по названию объявления (по `type_rules.yaml`);
  - проставляет `cabinet_name` (из параметра/окружения/конфига);
  - проставляет `redirect_target` по сотруднику (по `redirect_mapping.yaml`);
  - заполняет `avito_number_display` (ID объявления).
- Строит итоговую книгу с листами:
  - **Админ панель** — справочник объявлений (кабинет, название, регион/город, адрес, телефон/ID, переадресация);
  - **Диаграмма** — упрощённый набор данных для визуализации (кабинет, тип, название, адрес, ID объявления);
  - **Недельный** — помесячная/недельная детализация по объявлениям (просмотры, контакты и др.);
  - **Сводная** — агрегаты по неделям с полем «Период» в формате `ДД.ММ - ДД.ММ`.
- Применяет базовое форматирование:
  - порядок и имена листов;
  - жирная шапка и выравнивание по центру;
  - автофильтр на листе `Диаграмма`;
  - ширины колонок, совпадающие с `expected_report.xlsx` (low priority).

## Запуск pipeline (чтение → валидация → нормализация → генерация Excel)

Из корня проекта:

```bash
# (опционально) Создать минимальный input_example.xlsx, если своего файла ещё нет
python scripts/create_sample_input.py

# (рекомендуется) Задать кабинет по умолчанию
export DEFAULT_CABINET_NAME="Акрон Услуги"

# Запустить генерацию отчёта
# читает:  data/input_example.xlsx
# пишет:   data/output_report.xlsx
python scripts/run_report.py
```

Или из кода:

```python
from pathlib import Path
from app.application.services.report_generation import generate_report

content = Path("data/input_example.xlsx").read_bytes()
result = generate_report(content, "input_example.xlsx", cabinet_name="Акрон Услуги")
Path("data/output_report.xlsx").write_bytes(result.content_bytes)
```

## Входные файлы

- **Обязательный вход:** `data/input_example.xlsx`
  - один лист `Sheet1`;
  - заголовки в строке 1;
  - 30 колонок, соответствующих описанию в `data/input_structure.md`:
    - `Номер объявления`, `Регион размещения`, `Город`, `Адрес`, `Категория`, `Подкатегория`, `Параметр`,
      `Название объявления`, `Цена`, даты публикации/снятия, метрики показов/просмотров/контактов, расходы и т.д.
- **Эталонный выход:** `data/expected_report.xlsx`
  - используется только как пример целевого формата;
  - фактический `output_report.xlsx` максимально к нему приближен, но может отличаться по данным и сводной логике.

Источник истины по структуре входа/выхода: `data/input_structure.md`, `data/output_structure.md`.

## Структура проекта

- `app/application` — orchestration, исключения, статусы, DTO, сервис `generate_report(...)`.
- `app/domain` — валидация, нормализация, классификация, обогащение, расчёт сводной, сборка листов.
- `app/infrastructure` — Excel reader/writer/formatter, загрузка конфигов.
- `app/configs` — YAML:
  - `column_mapping.yaml` — маппинг входных колонок → канонические поля;
  - `type_rules.yaml` — правила определения типа объявления по названию;
  - `redirect_mapping.yaml` — маппинг сотрудник → куда переадресация;
  - `workbook_layout.yaml` — порядок листов, колонки, форматирование (в т.ч. ширины);
  - `input_schema.yaml` — имя листа, обязательные колонки, `default_cabinet_name`.

## Ограничения и отличия от expected_report.xlsx

Полный список ограничений зафиксирован в `data/KNOWN_LIMITATIONS.md`. Кратко:

- **Телефон:** во входном файле нет отдельной колонки с телефоном. В expected в колонке «Номер на Авито» используется телефон, в нашем отчёте — ID объявления (`avito_number_display`). Мы **не подставляем вымышленные телефоны**.
- **Число строк:** в expected на листах `Диаграмма` и `Недельный` строк больше, чем в текущем output (175/164 vs 160). Это связано с другим источником данных/периодом у эталонного файла, а не с ошибкой pipeline.
- **Сводная:** в expected две строки с периодами `16.02 - 22.02` и `23.02 - 01.03` и вручную заполненными комментариями. В нашем MVP:
  - периоды считаются автоматически **по неделям** на основе `first_publish_date`;
  - комментарий всегда пуст (заполняется вручную после генерации);
  - количество строк сводной зависит от объёма входных данных.
- **Формулы и гиперссылки:** в MVP основные метрики считаются в Python, а не формулами Excel; гиперссылки в выходном файле пока не создаются, мы сохраняем текст и ID.
