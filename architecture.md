
Архитектура MVP для внутреннего инструмента обработки Excel-выгрузок из Авито
1. Название проекта
Avito Report BuilderВнутренний инструмент агентства для автоматической обработки Excel-выгрузок из Авито и формирования готового Excel-отчёта.

2. Краткое описание продукта
Система предназначена для автоматизации ручной обработки Excel-выгрузок из Авито.
Сейчас пользователь вручную:
* удаляет лишние столбцы,
* оставляет нужные метрики,
* переименовывает поля,
* добавляет листы,
* считает дополнительные показатели,
* приводит таблицу к удобному формату отчёта.
Нужна программа, которая:
1. принимает исходный Excel-файл выгрузки Авито;
2. автоматически применяет заранее заданные правила преобразования;
3. формирует итоговый Excel-файл в фиксированном формате отчёта.
Это внутренний инструмент для агентства, ориентированный на нескольких сотрудников.

3. Цель и ценность системы
Цель
Сократить время подготовки отчётов и убрать ручную рутинную обработку Excel-выгрузок.
Ценность
Система должна:
* стандартизировать отчётность;
* уменьшить число ручных ошибок;
* ускорить подготовку отчётов;
* позволить нескольким сотрудникам работать по одному сценарию;
* упростить дальнейшую разработку и поддержку.

4. Основные требования к MVP
Обязательные требования
* загрузка одного Excel-файла;
* обработка файла по фиксированным правилам;
* формирование одного итогового Excel-файла;
* важна корректная Excel-структура и оформление;
* итоговый формат отчёта всегда один;
* формат входной выгрузки всегда один.
Ограничения MVP
* без базы данных;
* без личных кабинетов;
* без сложной системы пользователей;
* без ролевой модели;
* без внешних интеграций;
* без хранения истории отчётов.

5. Формат продукта
Рекомендуемый формат MVP: простое внутреннее web-приложение.
Сценарий использования
1. сотрудник открывает страницу;
2. загружает Excel-файл выгрузки Авито;
3. нажимает кнопку обработки;
4. получает готовый Excel-файл на скачивание.
Почему выбран web-формат
* инструментом будут пользоваться несколько сотрудников;
* web-формат удобнее, чем запуск локального скрипта;
* быстрее и проще реализовать, чем desktop-приложение;
* удобно развивать в будущем.

Для MVP
* Python 3.11+
* pandas — чтение, трансформация, агрегация табличных данных
* openpyxl — запись Excel-файла и оформление листов
* Streamlit — простой web-интерфейс для загрузки и скачивания файлов
* pytest — тесты
* ruff — линтинг / форматирование
* pydantic или dataclass — для конфигов и моделей данных

Почему этот стек
* быстро собирается MVP;
* хорошо подходит для работы с Excel;
* удобно поддерживать;
* удобно развивать;
* хорошо подходит для разработки через Cursor.

7. Границы системы
Система делает
* принимает один Excel-файл;
* валидирует его структуру;
* читает исходные данные;
* нормализует данные;
* применяет бизнес-правила трансформации;
* вычисляет производные поля;
* строит итоговые листы отчёта;
* оформляет итоговую Excel-книгу;
* отдает результат пользователю.
Система не делает
* не хранит отчёты;
* не хранит историю загрузок;
* не управляет пользователями;
* не позволяет редактировать правила через интерфейс;
* не загружает данные из Авито напрямую;
* не работает с несколькими входными файлами одновременно;
* не строит аналитику на основе прошлых периодов.

8. Пользователи и роли
Пользователи
Сотрудники агентства, которые готовят и используют отчёты по рекламе на Авито.
Роли в MVP
Ролей нет.Все пользователи работают одинаково:
* загружают файл;
* запускают обработку;
* скачивают результат.

9. Основной пользовательский сценарий
Базовый сценарий
1. Пользователь открывает приложение.
2. Загружает .xlsx-файл выгрузки Авито.
3. Система проверяет корректность файла.
4. Система применяет правила обработки.
5. Система формирует итоговую Excel-книгу.
6. Пользователь скачивает итоговый .xlsx.

* отсутствует нужный лист;
* отсутствуют обязательные колонки;
* файл пустой;
* файл поврежден.

14. Бизнес-правила
14.1 Общие правила
* входной формат всегда один;
* выходной формат всегда один;
* система работает только с одним Excel-файлом за раз;
* логика строго шаблонная, не универсальный конструктор.
14.2 Правила нормализации
* проверить наличие ожидаемого листа;
* проверить наличие обязательных колонок;
* удалить пустые строки;
* привести числовые поля к числам;
* очистить текстовые поля от лишних пробелов;
* привести структуру к внутренним именам колонок.
14.3 Правила формирования поля Тип
Поле Тип определяется по заголовку / названию объявления.
Примеры:
* если заголовок относится к теме аккумуляторов, например"Сдать старый аккумулятор. Приём аккумуляторов"то Тип = АКБ
* если заголовок относится к общему направлению, например"Прием и вывоз металлолома СПБ и ЛО"то Тип = Общее
Важно:
* правила классификации должны быть вынесены в отдельный модуль;
* желательно хранить их в конфиге или таблице правил, а не хардкодить хаотично по коду;
* нужно предусмотреть вариативность формулировок заголовков.
14.4 Правила формирования поля Кабинет Авито
Поле Кабинет Авито не берется напрямую из исходной выгрузки и должно задаваться вручную.
Для MVP допустимы варианты:
* константа в конфиге;
* ввод значения в интерфейсе перед генерацией;
* заполнение в итоговом Excel после генерации.
Архитектурно лучше предусмотреть отдельный источник этого значения.
14.5 Правила формирования поля Номер на Авито
Поле Номер на Авито берется из оригинальной таблицы, из поля с номером объявления.
14.6 Правила формирования поля Куда переадресация
Поле Куда переадресация связано с сотрудником и должно заполняться по данным, которые есть в оригинальной таблице или сопоставляются с полем сотрудника.
Это правило нужно реализовать как явное сопоставление поля источника к полю итогового листа.
14.7 Правила формирования листа Сводная
Все значения листа Сводная считаются только из текущего входного файла, без подтягивания истории и без прошлых периодов.
Примеры:
* просмотры = сумма просмотров по всем строкам;
* контакты = сумма контактов;
* чаты = сумма значений по чату;
* расходы = сумма расходов;
* и так далее по нужным метрикам.
Производные показатели:
* конверсия = сумма контактов * 100 / сумма просмотров
* средняя стоимость заявки = сумма расходов / сумма контактов
Поле Комментарий в Сводная должно оставаться пустым, чтобы специалист заполнял его вручную после генерации.

15. Архитектурный подход
Рекомендуемый стиль: modular monolith.
То есть это одно приложение, но логически разделенное на слои и модули.
Рекомендуемые слои
* UI — интерфейс загрузки/скачивания
* Application — orchestration / сценарий обработки
* Domain — бизнес-правила и трансформации
* Infrastructure — чтение/запись Excel, форматирование файлов
Почему это оптимально
* достаточно просто для MVP;
* удобно поддерживать;
* удобно тестировать;
* легко масштабировать позже;
* можно заменить интерфейс без переписывания ядра.

16. Предлагаемый pipeline обработки
Шаг 1. Upload file
Пользователь загружает Excel-файл.
Шаг 2. Validate workbook structure
Система проверяет:
* что файл Excel;
* что файл не пустой;
* что есть нужный лист;
* что есть обязательные колонки.
Шаг 3. Read source sheet
Система читает исходный лист в DataFrame.
Шаг 4. Normalize headers and values
* нормализует названия колонок;
* удаляет пустые строки;
* приводит типы данных;
* очищает текстовые значения.
Шаг 5. Build base dataframe
Формирует единый внутренний набор данных.
Шаг 6. Enrich dataframe
Добавляет вычисляемые и служебные поля:
* type
* cabinet_name
* redirect_target
* и другие derived fields
Шаг 7. Build output sheets
Строит отдельные наборы данных:
* для Админ панель
* для Диаграмма
* для Недельный
* для Сводная
Шаг 8. Write Excel workbook
Создает итоговую Excel-книгу.
Шаг 9. Apply formatting
Применяет:
* порядок листов;
* шапки;
* ширины колонок;
* фильтры;
* freeze panes;
* форматы чисел;
* другие элементы оформления.
Шаг 10. Return file
Возвращает файл пользователю на скачивание.

17. Нефункциональные требования
Простота
MVP должен быть максимально простым в запуске и использовании.
Поддерживаемость
Код должен быть разделен по слоям и модулям, чтобы правила обработки можно было менять без переписывания всего проекта.
Надежность
Система должна стабильно обрабатывать стандартную выгрузку Авито.
Производительность
Обработка одного Excel-файла должна занимать секунды.
Расширяемость
Архитектура должна позволять:
* добавлять новые листы;
* изменять правила;
* дорабатывать классификацию Тип;
* заменить Streamlit на API/UI в будущем.

18. Обработка ошибок и нестандартных ситуаций
Система должна обрабатывать:
Ошибки файла
* загружен не Excel;
* файл пустой;
* файл поврежден.
Ошибки структуры
* отсутствует нужный лист;
* отсутствуют обязательные колонки;
* неожиданные заголовки колонок.
Ошибки данных
* пустые числовые значения;
* строковые значения вместо чисел;
* деление на ноль в производных метриках;
* частично неполные строки.
Рекомендуемое поведение
* критические ошибки структуры: остановить обработку и показать понятное сообщение;
* некритичные проблемы данных: по возможности продолжить обработку, подставляя безопасные значения;
* деление на ноль: возвращать 0, пустое значение или другое заранее согласованное безопасное значение.

20. Назначение ключевых модулей
ui/streamlit_app.py
Отвечает за:
* загрузку файла;
* запуск обработки;
* отображение ошибок;
* скачивание результата.
application/report_service.py
Главный orchestration-слой.Должен содержать основную функцию, например:
generate_report(input_bytes: bytes, cabinet_name: str | None = None) -> bytes
Эта функция:
* валидирует вход;
* читает Excel;
* нормализует данные;
* добавляет derived fields;
* строит листы;
* формирует итоговый workbook;
* возвращает готовый файл.
domain/validators.py
Проверяет:
* тип файла;
* наличие нужного листа;
* наличие обязательных колонок;
* минимальную корректность структуры.
domain/classifiers.py
Содержит логику определения поля Тип по заголовку объявления.
domain/transformers.py
Содержит:
* очистку данных;
* переименование колонок;
* приведение типов;
* преобразование к внутренней схеме;

domain/sheet_builders.py
Формирует данные листов:
* build_admin_sheet(...)

domain/summary_calculator.py
Формирует данные для листа Сводная:

infrastructure/excel_reader.py
Читает исходный Excel-файл.
infrastructure/excel_writer.py
Создает итоговую Excel-книгу и записывает в нее данные листов.
infrastructure/workbook_formatter.py
Применяет оформление:
* ширины колонок;
* стили;
* фильтры;
* freeze panes;

Что можно вынести в конфиг
* обязательные входные колонки;
* сопоставление имен колонок;
* порядок колонок в каждом листе;
* layout листов;
* правила классификации Тип;

Зачем это нужно
* проще править без переписывания логики;
* легче поддерживать;
* удобнее тестировать;
* меньше хрупкого хардкода.

22. Рекомендации по реализации расчётов
Что лучше считать в Python
Для MVP желательно считать основные метрики в Python, а в Excel записывать уже готовые значения.
Это касается:
* сумм;
* конверсий;
* средней стоимости заявки;
* других агрегированных показателей.
Почему
* проще тестировать;
* меньше зависимости от Excel-формул;
* меньше риска сломать расчёты при ручном редактировании;
* предсказуемый результат.
Что можно оставить как Excel-формулы
Только то, что действительно должно быть доступно для ручной правки или прозрачного просмотра пользователем.

23. Риски и архитектурные замечания
1. Скрытые бизнес-правила
Основной риск — наличие дополнительных ручных исключений, которые не были формализованы заранее.
2. Классификация по заголовку
Поле Тип может требовать набора правил, синонимов и исключений, потому что заголовки объявлений могут сильно различаться.
3. Excel-оформление
Excel-форматирование нельзя смешивать с бизнес-логикой. Это отдельный слой.
4. Ручные поля
Поля вроде Кабинет Авито нужно заранее формализовать как источник значения:
* константа,
* ввод в UI,
* или постобработка.
5. Хрупкость к изменению шаблона Авито
Если Авито изменит структуру выгрузки, понадобится корректировать слой валидации и маппинг колонок.

24. Что обязательно должно быть в MVP
* web-интерфейс загрузки файла;
* валидация входного Excel;
* чтение и нормализация данных;
* классификация Тип;
* формирование листов:
    * Админ панель
    * Диаграмма
    * Недельный
    * Сводная
* расчёт сводных показателей;
* экспорт итогового Excel;
* оформление итогового workbook;
* понятные ошибки пользователю.

25. Что можно отложить на вторую очередь
* несколько типов шаблонов отчётов;
* редактирование правил через интерфейс;
* хранение истории генераций;
* автоматическое заполнение ручных полей по справочникам;
* API;
* Docker и развертывание во внутренней сети;
* логирование действий пользователей;
* сохранение файлов;
* интеграции с внешними системами;
* сравнение периодов;
* мультифайловый режим.

26. Рекомендуемый архитектурный вывод
Для этой задачи оптимальна следующая архитектура:
Python modular monolith + Streamlit UI + pandas/openpyxl + конфигурация правил в YAML
Это решение:
* простое;
* реалистичное;
* быстрое в разработке;
* удобное для внутреннего использования;
* подходит для нескольких сотрудников;
* хорошо масштабируется в будущем;
* подходит для реализации через Cursor.

27. Итоговая рекомендация для разработчика
Систему нужно реализовывать как шаблонный генератор одного конкретного отчёта, а не как универсальный конструктор Excel.
Приоритеты разработки:
1. сначала ядро обработки данных;
2. затем построение листов;
3. затем экспорт и форматирование Excel;
4. затем UI;
5. затем тесты и шлифовка правил.

28. Что уже достаточно определено для старта разработки
Достаточно определено:
* формат продукта;
* технологический стек;
* границы системы;
* входной и выходной форматы;
* основные листы;
* базовые бизнес-правила;
* логика сводной;
* принцип определения Тип;
* отсутствие БД и ролей;
* важность Excel-оформления.

29. Что нужно формализовать перед написанием кода
Перед написанием кода нужно отдельно зафиксировать:
* точный список обязательных колонок входной таблицы;
* точное сопоставление входных колонок и колонок каждого листа;
* полный набор правил классификации Тип;
* точное правило заполнения Куда переадресация;
* точный способ заполнения Кабинет Авито;
* точные требования к Excel-оформлению по каждому листу.
